# `explorer/` — local language

The teach loop. Proactively crawls a site to populate a SiteGraph **before**
any agent uses it.

## Intent in one paragraph

`explorer/` pays the LLM exploration cost **once per site** so that future
agents can skip it. It opens a real browser, lets the user log in, then
walks the site (depth-bounded, time-bounded, throttled), recording states
and the transitions between them. The output is a SiteGraph rich enough that
`runtime/` can navigate the site by graph traversal instead of LLM-driven
clicks. This is the cache-warming half of the project; without it we're just
Stagehand-with-a-graph-format.

## Local vocabulary

### teach loop
The user-facing sequence. `siteforge teach <url>` runs the seven-phase pipeline
and persists results.

### phase
A single named step in the pipeline DAG. Phases: `SETUP`, `SNAPSHOT`,
`CLASSIFY`, `ENUMERATE`, `VALIDATE`, `CLUSTER`, `TRACE`. Each consumes the
previous phase's output and produces typed data for the next.

### candidate operation
A potential `Operation` generated by the `ENUMERATE` phase but not yet
executed. Confidence 0.7, reason `llm-inferred-untested`. Becomes a real
Operation only after `VALIDATE` runs it and observes a state transition.

### frontier
The set of `State`s discovered but not yet fully explored. `VALIDATE` pulls
from frontier, expands one State, pushes newly-discovered neighbors back on.
Bounded by `max_depth`.

### anchor state
A "safe" state we know how to navigate back to (typically the home page).
Used by the explorer when `page.goBack()` won't work (e.g. modal dismissal).
Configured manually in v1; auto-detected later.

### backtrack
After validating one candidate operation from a state, the explorer must
return to that state to test other candidates. v1 strategy: `page.goBack()`,
fall back to `goto(anchor_url)`. Some sites break on back; the anchor fallback
is the safety net.

### throttle
The randomized delay between operations during `VALIDATE` (default 1-3s,
jittered). Anti-detection. Enforced even on fast hardware. Min 1s.

### single-flight
A locking primitive on `(domain, state_id, op_id)` that prevents two parallel
explorer runs (or one run with concurrent branches) from re-exploring the
same node. Second caller waits for the first.

### entry-point score
A heuristic 0..100 score assigned to each State during `TRACE`. Inputs: URL
matches `/`/`dashboard`, role contains `main`, high out-degree, low in-degree.
Top-N entry points become starting points for process tracing.

### dangerous click
An operation we deliberately skip during `VALIDATE`: form submits with
non-empty inputs, send-DM buttons, post-publish buttons. Heuristic: button
nested in a `form` role with at least one filled input → skip. Configurable
per site.

## Not in this module

- ❌ Executing user intents (that's `runtime/runProcess`)
- ❌ Path planning at run time (that's `core/planPath` via `runtime/`)
- ❌ Emitting CLI binaries (that's `emitters/`)
- ❌ Defining what an Atom is (that's `core/types` + `snapshot/`)
- ❌ Storing the graph (that's `storage/saveGraph` — explorer just calls it)

## Key invariants

1. **Bounded work.** Every phase respects `time_budget_ms`. The explorer is
   never an infinite crawler.
2. **Resumable.** If killed mid-`VALIDATE`, the partial graph is saved; the
   next `explore()` call picks up from where it left off.
3. **Idempotent.** Running `explore()` twice on the same domain extends the
   graph but never duplicates states or operations.
4. **Anti-detection-aware.** Throttling is enforced. No fast loops. No
   parallel clicks. No actions that look obviously robotic.
5. **No silent destructive actions.** Dangerous clicks are skipped, not
   executed. We err on the side of "didn't discover this transition" over
   "accidentally posted a comment".
